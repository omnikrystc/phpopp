Since these are all so interrelated I am going to wrap them all in one project and make notes here...

Design Pattern: Registry (includes/edp/base/registry.php)

  • An expansion on the Singleton pattern used to centralize common objects/functions (AKA globals).
  • Divided into three classes to separate scope (request, session, application)
  • Normally would use the request and either session or application but not both

Design Pattern: Command (includes/edp/command/Command.php)

  • This has actually been covered in its own demo but is used heavily here.
  • The CommandResolver (includes/edp/command/commandresolver.php) serves as a command factory for the Front Controller. This is a basic solution that directly links commands to views. This is fine for small systems but not flexible enough for larger or more complex systems.

Design Pattern: Front Controller (includes/edp/controller/controller.php)

  • While the controller itself is fairly basic it will most likely need helper classes to support it.
  • CommandResolver is a Command factory for the Controller
  • Request (includes/edp/controller/request.php) wraps the PHP globals and allows us control over the data.
  • ApplicationHelper (includes/edp/controller/applicationhelper.php) provides access to configuration data.

Design Pattern: Application Controller (includes/edp/controller/controller.php)

  • Replaces the CommandResolver allowing multiple commands to be executed and more dynamic view logic.
